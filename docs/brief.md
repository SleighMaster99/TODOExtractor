# Project Brief: Todo Extractor

**작성 상태:** ✅ 완료
**마지막 업데이트:** 2025-11-23

---

## Executive Summary

**Todo Extractor**는 Visual Studio 2022 확장 프로그램으로, 개발자가 코드에 작성한 TODO 주석을 자동으로 추출하여 GitHub/GitLab 이슈로 변환하는 도구입니다.

**핵심 문제:** 개발자들은 코딩 중 떠오르는 작업을 `//TODO:` 형태로 주석에 남기지만, 이를 체계적으로 추적하고 이슈화하는 과정이 수동적이고 번거롭습니다.

**대상 시장:** Visual Studio 2022를 사용하는 개발자 및 개발팀

**핵심 가치:** 사용자 정의 마커 기반 주석 추출과 GitHub/GitLab 자동 이슈 생성을 통해 코드 내 작업 항목을 효율적으로 관리하고 추적할 수 있습니다.

---

## Problem Statement

**현재 상태와 문제점:**

개발자들은 코딩 중 다음과 같은 작업을 주석으로 남깁니다:
- 추후 구현해야 할 기능
- 리팩토링이 필요한 부분
- 버그 수정 사항
- 성능 개선 항목
- 문서화가 필요한 영역

하지만 이러한 주석들은:
- 코드베이스 전체에 흩어져 있어 관리가 어렵습니다
- 수동으로 찾아서 이슈 트래커에 등록해야 하므로 시간이 소요됩니다
- 개발 흐름을 방해하지 않으려면 나중으로 미루게 되고, 결국 잊혀집니다
- 팀 차원에서 우선순위를 관리하거나 할당하기 어렵습니다

**문제의 영향:**

- **생산성 저하:** 개발자가 주석을 수동으로 찾고 이슈로 옮기는 데 시간을 낭비합니다
- **작업 누락:** 중요한 작업이 코드 주석에 묻혀 실행되지 않습니다
- **추적 불가:** 남겨진 작업의 전체 현황과 우선순위를 파악할 수 없습니다
- **협업 어려움:** 팀원 간 작업 항목 공유와 할당이 비효율적입니다

**기존 솔루션의 한계:**

- **수동 검색:** VS의 "Find in Files"로 TODO를 찾아도 이슈화는 별도 작업
- **기존 확장 프로그램:** Task List 등은 단순 나열만 제공하며 이슈 관리 시스템과 연동 부족
- **외부 스크립트:** grep 등 CLI 도구는 VS 환경과 통합되지 않아 개발 흐름 단절

**해결의 시급성:**

현대 애자일 개발에서 작업의 가시성과 추적성은 필수입니다. 코드 내 작업 항목을 즉시 이슈 트래커로 전환할 수 있어야 진정한 지속적 개선이 가능합니다.

## Proposed Solution

**핵심 컨셉:**

Todo Extractor는 Visual Studio 2022에 긴밀하게 통합된 확장 프로그램으로, 개발자의 코드 내 작업 주석을 자동으로 발견하고 이슈 관리 시스템으로 변환하는 솔루션입니다.

**작동 방식:**
1. 사용자가 커스텀 마커(예: `TODO:`, `FIXME:`, `OPTIMIZE:`)를 정의
2. 확장 프로그램이 지정된 경로의 코드를 스캔하여 `//마커` 패턴의 주석 추출
3. 추출된 항목을 그룹화, 필터링, 우선순위화
4. GitHub/GitLab API를 통해 자동으로 이슈 생성 또는 로컬 파일로 저장

**기존 솔루션 대비 차별화 요소:**

- **완전한 커스터마이제이션:** 사용자 정의 마커, 중요도 설정, 다중 마커 지원
- **양방향 출력:** 이슈 트래커 직접 연동과 로컬 파일 저장(MD, CSV, JSON, HTML, DB) 모두 지원
- **Visual Studio 네이티브 통합:** IDE 워크플로우를 벗어나지 않고 사용 가능한 세련된 UI
- **스마트 필터링:** 파일 확장자, 디렉터리 제외, 경로 지정 등 정교한 스캔 제어
- **미리보기 기능:** 이슈 생성 전 검토 및 선택적 실행 가능

**성공 요인:**

- **개발자 워크플로우 존중:** 코딩 흐름을 방해하지 않고 자연스럽게 통합
- **즉각적인 가치:** 설치 후 바로 사용 가능한 직관적 인터페이스
- **유연성:** 개인 개발자부터 팀까지 다양한 사용 사례 지원
- **오픈 생태계:** 주요 이슈 트래커(GitHub, GitLab)와 표준 파일 형식 지원

**제품 비전:**

개발자가 "나중에 해야지"라고 생각하는 순간, 그 생각을 코드 주석으로 남기면 자동으로 추적 가능한 작업 항목이 되는 세상을 만듭니다. 코드와 작업 관리 사이의 마찰을 제거하여 진정한 지속적 개선 문화를 촉진합니다.

## Target Users

### Primary User Segment: 개인 개발자

**프로필:**
- VS 2022를 주 개발 도구로 사용하는 .NET/C# 개발자
- 개인 프로젝트 또는 소규모 팀(1-5명)에서 작업
- GitHub 또는 GitLab을 이슈 트래커로 사용
- 연령대: 25-45세, 경력 2-15년

**현재 행동 및 워크플로우:**
- 코딩 중 `//TODO:`, `//FIXME:` 등의 주석을 습관적으로 작성
- 주기적으로 "Find in Files"로 TODO를 검색하거나, 전혀 검색하지 않음
- 이슈 트래커에 수동으로 작업 항목 등록 (시간이 날 때)
- 중요한 작업이 주석에 묻혀 잊혀지는 경험

**구체적 니즈와 고충:**
- 코딩 흐름을 끊지 않고 작업 항목을 빠르게 기록하고 싶음
- 나중에 할 일을 잊지 않고 체계적으로 관리하고 싶음
- 수동 복사-붙여넣기 작업 없이 주석을 이슈로 변환하고 싶음
- 프로젝트 전체의 남은 작업 현황을 한눈에 보고 싶음

**달성하려는 목표:**
- 기술 부채 최소화 및 코드 품질 향상
- 프로젝트 진행 상황의 투명한 추적
- 작업 우선순위화 및 계획 수립
- 개발 생산성 향상

### Secondary User Segment: 팀 리드 / 프로젝트 관리자

**프로필:**
- 3-20명 규모 개발팀의 리드 또는 PM
- 팀원들의 코드와 작업 진행을 관리
- 스프린트 계획 및 백로그 관리 책임
- 연령대: 30-50세, 경력 5년 이상

**현재 행동 및 워크플로우:**
- 코드 리뷰 중 TODO 주석 발견 시 팀원에게 이슈 생성 요청
- 정기적으로 팀의 미완료 작업 현황 파악 시도
- Jira, GitHub Issues 등에서 백로그 관리
- 기술 부채 항목 추적의 어려움 경험

**구체적 니즈와 고충:**
- 팀원들이 남긴 TODO 주석의 가시성 확보
- 코드베이스 전체의 기술 부채 현황 파악
- 팀원별 남은 작업 분포 확인
- 우선순위별 작업 항목 자동 분류

**달성하려는 목표:**
- 팀의 작업 투명성 및 책임감 향상
- 스프린트 계획 시 숨겨진 작업 항목 반영
- 기술 부채 관리 및 리팩토링 계획 수립
- 팀 생산성 및 코드 품질 지표 개선

## MVP Scope

### Core Features (Must Have)

- **커스텀 마커 관리:** 사용자가 여러 마커(예: TODO, FIXME, OPTIMIZE)를 등록, 편집, 삭제할 수 있으며 각 마커에 중요도(높음/보통/낮음) 설정 가능. 마커 중복 방지 유효성 검사 포함

- **주석 추출 엔진:** `//마커` 패턴으로 시작하는 주석만 추출. 지정된 파일 경로를 스캔하며 파일 확장자 필터(.cs, .cpp, .js 등)와 제외 디렉터리(bin, obj, node_modules) 설정 지원

- **GitHub/GitLab 자동 이슈 생성:** API 토큰 기반 인증으로 추출된 TODO를 자동으로 이슈로 생성. 저장소 URL, 기본 라벨, 담당자 지정 설정 지원

- **로컬 파일 저장:** 추출 결과를 Markdown, CSV, JSON, HTML, DB(SQLite) 형식으로 저장. 그룹화 방식(마커별/파일별/우선순위별) 및 정렬 순서 선택 가능

- **결과 미리보기 UI:** 이슈 생성 또는 파일 저장 전 추출된 항목을 확인하고 선택적으로 실행할 수 있는 인터페이스. 일괄 선택/해제, 검색/필터 기능 포함

- **설정 관리 UI:** 마커 설정, 검사 경로 설정, 출력 옵션, 이슈 트래커 연동 설정을 관리하는 직관적인 UI

### Out of Scope for MVP

- 빌드 시 자동 스캔 기능
- Jira, Azure DevOps 등 다른 이슈 트래커 연동
- 팀 협업 기능 (공유 설정, 통계 대시보드)
- AI 기반 TODO 우선순위 자동 추천
- VS Code, JetBrains IDE 등 다른 IDE 지원
- 실시간 TODO 감지 및 알림
- 모바일 앱 또는 웹 대시보드

### MVP Success Criteria

MVP가 성공했다고 판단하는 기준:

- 100명 이상의 초기 사용자 확보 (첫 2개월)
- 평균 평점 3.5 이상 달성
- 사용자가 설치 후 첫 실행 시 5분 내에 첫 이슈 생성 성공률 60% 이상
- 치명적 버그 월 5건 미만
- 사용자 피드백을 통해 다음 개발 우선순위 명확화
- 최소 3개 이상의 긍정적 사용 사례/후기 확보

## Post-MVP Vision

### Phase 2 Features

MVP 출시 후 사용자 피드백을 바탕으로 우선순위를 결정할 예정이며, 다음 기능들을 고려하고 있습니다:

- **AI 기반 이슈 자동 정리 및 제목 생성 (우선순위 최상위):** 추출된 TODO 리스트를 AI가 분석하여 유사한 내용은 통합하고, 너무 광범위한 내용은 분리. 각 이슈에 대해 맥락을 이해하고 명확하고 실행 가능한 이슈 제목을 자동 생성. MVP 1.0 직후 버전에 포함 예정

- **빌드 시 자동 스캔:** CI/CD 파이프라인 통합으로 커밋 또는 빌드 시 자동으로 TODO 추출 및 이슈 생성

- **Jira 및 Azure DevOps 연동:** 엔터프라이즈 팀을 위한 추가 이슈 트래커 지원

- **팀 협업 기능:** 공유 가능한 마커 설정, 팀 대시보드, TODO 통계 및 트렌드 분석

- **고급 필터링 및 검색:** 날짜 범위, 작성자별, Git blame 정보 통합

- **TODO 템플릿 기능:** 자주 사용하는 TODO 패턴을 템플릿으로 저장하고 재사용

- **마커별 커스텀 액션:** 특정 마커에 대해 자동 라벨 추가, 담당자 할당 등 세부 규칙 설정

**비전:**

- 개발자가 "나중에 해야지"라는 생각을 절대 잊지 않는 환경
- 팀의 기술 부채를 실시간으로 추적하고 관리할 수 있는 생태계

### Platform Requirements

- **Minimum VS Version:** Visual Studio 2022 17.0 이상
- **Performance Requirements:**
  - 10,000개 파일 프로젝트에서 스캔 시간 30초 이내
  - UI 응답 시간 1초 이내
  - 메모리 사용량 200MB 이하

### Technology Preferences

- **Frontend (VS Extension UI):**
  - WPF (Windows Presentation Foundation) - VS 확장 표준
  - XAML을 통한 UI 디자인
  - MVVM 패턴 적용

- **Backend (Extension Logic):**
  - C# (.NET Framework 4.8 또는 .NET 6+)
  - Visual Studio SDK
  - Roslyn API (코드 분석용, 선택적)

- **Database:**
  - SQLite (로컬 설정 저장 및 DB 출력 형식 지원)
  - 설정 파일은 JSON 형식으로도 지원

- **External APIs:**
  - GitHub REST API v3 / GraphQL API
  - GitLab REST API v4
  - Octokit.NET (GitHub .NET SDK)

### Architecture Considerations

- **Repository Structure:**
  - 단일 리포지토리 (모놀리식)
  - 확장 프로그램 프로젝트 + 단위 테스트 프로젝트

- **Service Architecture:**
  - 플러그인 아키텍처로 설계하여 향후 이슈 트래커 추가 용이
  - 인터페이스 기반 설계 (IIssueTracker, IFileExporter 등)
  - 비동기 처리 (async/await) 활용

- **Integration Requirements:**
  - Visual Studio의 Solution Explorer와 통합
  - VS 설정 시스템과 연동
  - 출력 창(Output Window) 활용

- **Security/Compliance:**
  - API 토큰은 Windows Credential Manager에 암호화 저장
  - HTTPS 통신만 허용
  - 민감 정보 로깅 금지
  - GDPR 준수 (사용자 데이터 최소 수집)

- 초기 사용자는 주로 개인 개발자이며, 팀 기능은 Phase 2에서 수요가 증가할 것으로 가정
- VS Marketplace를 통한 배포가 주요 유통 채널이 될 것으로 가정

## Risks & Open Questions

### Key Risks

- **VS SDK 호환성 문제:** Visual Studio SDK의 향후 변경으로 인한 확장 프로그램 호환성 깨짐. 각 VS 버전 업데이트 시 테스트 및 수정 필요

- **API Rate Limiting:** GitHub/GitLab API 사용 제한으로 인한 대량 이슈 생성 실패. 특히 대규모 프로젝트에서 수백 개의 TODO를 한 번에 처리할 때 문제 발생 가능

- **사용자 온보딩 장벽:** API 토큰 생성 및 설정이 기술적으로 익숙하지 않은 사용자에게 진입 장벽이 될 수 있음. 자세한 가이드 및 튜토리얼 필요

- **성능 이슈:** 대규모 프로젝트(10,000개 이상 파일)에서 스캔 시간이 과도하게 길어질 수 있음. 최적화 및 인덱싱 전략 필요

- **경쟁 제품 출현:** 유사한 기능을 제공하는 다른 확장 프로그램이 출시될 위험. 차별화된 기능(AI 통합, 다중 출력 형식 등)으로 대응 필요

### Open Questions

- DB 출력 시 어떤 데이터베이스를 지원할 것인가? (SQLite, SQL Server, MySQL 등)
- GitHub와 GitLab 외 다른 이슈 트래커(Jira, Azure DevOps) 지원 우선순위는?
- 대소문자 구분 처리 방식은? (TODO vs todo vs Todo)
- 다국어 지원이 필요한가? (UI 및 문서)
- 멀티라인 TODO 주석을 어떻게 처리할 것인가?
- 이미 생성된 이슈와의 중복 방지 메커니즘이 필요한가?
- 무료 버전과 유료 버전(프리미엄 기능)을 구분할 것인가?

### Areas Needing Further Research

- VS 2022 확장 프로그램 개발 모범 사례 및 성능 최적화 기법
- GitHub/GitLab API 제한 회피 전략 및 배치 처리 방법
- 사용자 경험 향상을 위한 경쟁 제품 분석 (Task List, CodeMaid 등)
- AI 기반 이슈 제목 생성을 위한 적절한 모델 및 API 선택
- 대규모 코드베이스에서의 효율적인 파일 스캐닝 알고리즘
- VS Marketplace 마케팅 및 사용자 획득 전략
- 오픈소스 vs 상용 라이선스 결정
